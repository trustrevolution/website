# Trust Revolution Website Architecture

**Version**: 1.0  
**Date**: December 25, 2024  
**Stack**: Hugo + Netlify  
**Timeline**: Launch ASAP (2-3 weeks)

---

## OVERVIEW

Build a minimal, content-first podcast website at trustrevolution.co to:
- Own episode content and permalinks (protection against platform changes)
- Serve as content hub and launch point to Fountain (where sat streaming works)
- Collect email subscribers via Buttondown
- Support future written content (posts, research, analysis)
- Maintain clean, scannable mobile-first design matching shawnyeager.com aesthetic

**Current state**: trustrevolution.co redirects to Fountain  
**Target state**: Full Hugo site with automated episode updates and prominent Fountain CTAs

**Design Philosophy**: Site is for reading (show notes, guest info, written content) and launching to listening platforms (primarily Fountain for sat streaming support). No embedded players - every play should drive value-for-value monetization.

**Visual Identity**: Bold, provocative, revolutionary. High contrast, sharp edges, aggressive typography. Think brutalist web design meets punk zines - not polite magazine layouts.

## CONTENT ARCHITECTURE

### Episode Data Sources

**Source 1: Fountain RSS Feed (automated via GitHub Action)**
- Episode title
- Publication date
- Audio URL (main enclosure)
- Video URL (alternate enclosure, if present)
- Duration
- Episode artwork URL
- Episode number (parsed from title or ID)

**Source 2: Show Notes Markdown (manual commit)**
- Full show notes content (hook, summary, guest bio, quotes, timestamps, resources)
- Guest metadata (name, social links, bio)
- Season/episode number
- Local artwork path (if different from Fountain)

**Content Workflow**:
1. Generate show notes with Claude (existing process)
2. Save markdown to: `content/episodes/s##e##.md`
3. Commit to Hugo repo
4. GitHub Action triggers on commit:
   - Parses Fountain RSS for latest episode metadata
   - Updates `data/latest_episode.json`
   - Merges RSS data with markdown frontmatter
5. Netlify auto-deploys

## LAUNCH-POINT STRATEGY

### Why No Embedded Players

**Value-for-Value Alignment**:
- Sat streaming only works in Fountain app/environment
- Every listen on-site without streaming = lost monetization opportunity
- Site playback would cannibalize Fountain engagement

**Technical Benefits**:
- Faster page loads (no video/audio assets)
- Simpler codebase (no player components)
- Lower bandwidth costs
- Better mobile performance

**User Experience**:
- Clear call-to-action (one primary path: Fountain)
- No confusion about where to listen
- Platform-appropriate experiences (Fountain = sats, YouTube = discovery, Spotify = convenience)

**Future Flexibility**:
- If Fountain launches embeddable sat-enabled player, easy to integrate
- Until then, site serves as high-quality content hub + launch point
- Can test embedded players later without architectural changes

### Content Strategy

**Site is for**:
- Reading show notes (searchable, ownable, SEO-optimized)
- Discovering guests (directory, bios, social links)
- Finding episodes (archive, filtering by season/guest)
- Reading written content (posts, analysis, research)
- Collecting email subscribers

**Fountain/Platforms are for**:
- Listening (sat streaming, comments, social features)
- Supporting (boosts, splits, direct payments)
- Engaging (clips, playlists, community)

## DESIGN SPECIFICATIONS

### Visual Identity: Revolution

**Design Philosophy**: Trust Revolution challenges systems, exposes broken institutions, and champions alternatives. The design should be **bold, provocative, and unapologetic** - not polite or magazine-clean. Think underground zine meets modern brutalism.

**Core Principles**:
- **Contrast over subtlety**: High contrast, sharp edges, clear hierarchy
- **Energy over elegance**: Movement, tension, visual disruption
- **Direct over polished**: Rough edges are features, not bugs
- **Impact over prettiness**: Every element should have purpose and punch

```yaml
name: Update Latest Episode Data

on:
  push:
    branches: [main]
    paths: 
      - 'content/episodes/**'
  schedule:
    # Run daily at 6 AM UTC to check for new Fountain episodes
    - cron: '0 6 * * *'

  update-episode-data:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout repo
        uses: actions/checkout@v3

      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'

      - name: Install dependencies
        run: npm install rss-parser

      - name: Fetch Fountain RSS and update data
        run: |
          node scripts/update-episode-data.js

      - name: Commit changes
        run: |
          git config user.name github-actions
          git config user.email github-actions@github.com
          git add data/latest_episode.json data/fountain_rss.json
          git diff --quiet && git diff --staged --quiet || \
            git commit -m "Auto-update episode data from Fountain RSS"
          git push
```

### Script: `scripts/update-episode-data.js`

```javascript
const Parser = require('rss-parser');
const fs = require('fs');
const path = require('path');

const FOUNTAIN_RSS_URL = 'https://feeds.fountain.fm/OIYZniSDb9jd3Pb78CpF';

async function updateEpisodeData() {
  const parser = new Parser({
    customFields: {
      item: [
        ['itunes:duration', 'duration'],
        ['itunes:image', 'image']
      ]
    }
  });

  try {
    // Fetch RSS feed
    const feed = await parser.parseURL(FOUNTAIN_RSS_URL);
    
    // Get latest episode
    const latestEpisode = feed.items[0];
    
    // Build data object (metadata only - no media URLs needed)
    const episodeData = {
      title: latestEpisode.title,
      pubDate: latestEpisode.pubDate,
      duration: latestEpisode.duration || null,
      artwork: latestEpisode.image?.href || latestEpisode.itunes?.image || null,
      link: latestEpisode.link,
      description: latestEpisode.contentSnippet || latestEpisode.description,
      updatedAt: new Date().toISOString()
    };
    
    // Write latest episode data
    fs.writeFileSync(
      path.join(__dirname, '../data/latest_episode.json'),
      JSON.stringify(episodeData, null, 2)
    );
    
    // Cache full RSS data
    fs.writeFileSync(
      path.join(__dirname, '../data/fountain_rss.json'),
      JSON.stringify(feed.items, null, 2)
    );
    
    console.log('Successfully updated episode data');
    
  } catch (error) {
    console.error('Error updating episode data:', error);
    process.exit(1);
  }
}

updateEpisodeData();
```

## TECHNICAL NOTES

### Fountain RSS Structure
Based on Podcasting 2.0 spec, Fountain's RSS includes:
- Main `<enclosure>`: Audio MP3
- `<podcast:alternateEnclosure>`: Video MP4 (if uploaded)
- `<itunes:image>`: Episode artwork
- `<itunes:duration>`: Episode length

**What we extract**: Duration, artwork, metadata only  
**What we skip**: Media URLs (not needed - no embedded players)

### Fountain Integration Strategy
- **RSS metadata only**: Keep homepage current, display episode info
- **Deep links**: Send listeners to Fountain app/web for playback
- **Sat streaming**: Only works in Fountain environment (app or their web player when fully launched)
- **Site purpose**: Content hub (show notes, guest info, posts) + launch point to Fountain

### Portability Plan
If migrating from Fountain in the future:
1. All MP3/MP4 files backed up locally
2. Upload to new host (Backblaze B2, Cloudflare R2, etc.)
3. Update Fountain episode URLs to new host URLs
4. GitHub Action pulls from new RSS source
5. Site structure unchanged (still launch point, just different destination)
